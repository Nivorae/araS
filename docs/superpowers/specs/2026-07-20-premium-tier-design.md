## 產品定位（決策的前提）

**araS 是資產記錄 App，不是記帳 App，也不是交易 App。**

這句話否決了好幾個一開始被提出的方向：

- **定期定額自動記帳**——記帳 App 的思維，與定位不合
- **儲蓄率／月支出健檢**——同上
- **即時報價作為付費點**——交易 App 的賣點，見下節

付費分層必須從「記錄資產」這件事出發。

## 核心決策

### 決策一：採用量限制 + 功能分層，不做 grandfather

考慮過的三種方向：

| 方向                              | 評估                                             |
| --------------------------------- | ------------------------------------------------ |
| A. 既有功能全免費，只有新功能收費 | 不激怒既有用戶，但要先做出夠份量的新東西才有得賣 |
| B. 把部分既有功能劃進付費         | 馬上有得賣，但既有用戶感覺功能被拿走             |
| **C. 用量上限（採用）**           | 對新用戶自然，對重度用戶轉換率高                 |

**不做 grandfather 機制**：上限訂 20 時只有 1 位用戶超過，為單一用戶建立永久豁免的資料欄位不划算，且會污染後續所有權限判斷邏輯。

### 決策二：免費上限為 20 筆資產

不是 15，理由：

1. **15 會誤傷核心用戶**——擋到 4 人（10.3%），而這 4 人正是最投入的使用者。那是流失，不是轉換。
2. **20 幾乎不誤傷**——只有 1 人超過（22 筆）。
3. **20 卡在正確的位置**——台灣用戶的自然筆數（銀行 3–5、信用卡／貸款 2–3、股票 5–10、保單 2–4、現金 1–2）約落在 15–25。碰到上限的人，正是願意付錢的人。
4. **心理感受**——「20 筆」讀起來大方，「15 筆」讀起來小氣，但兩者擋到的付費族群（30 筆以上）相同。

### 決策三：超過上限時「只擋新增」

超額用戶的既有資料**完全正常**：照看、照編輯、照計入圖表與淨值。只有新增第 21 筆時出現升級提示。

替代方案「唯讀降級」（鎖住超額資料）被否決——資料看得到卻不能動，體感是懲罰，對唯一那位 22 筆的用戶（全站最投入者）尤其不該如此。

### 決策四：報價、匯率完全免費，不設限

**這是從原本方向反轉的決定，理由如下。**

原先假設報價是「成本型功能」（免費用戶越多越虧），實際查證後不成立：

```ts
// apps/web/services/quotes.service.ts
const QUOTE_CACHE_SECONDS = 30; // Next.js Data Cache，跨 serverless 實例共享
```

主來源是 Yahoo 非官方 chart API——免金鑰、零費用；備援 Finnhub／CoinGecko 走免費額度。30 秒共享快取代表 39 位或 39,000 位用戶同時查同一檔股票，對上游都只是每 30 秒一次請求。**成本不是限制因素。**

曾考慮「免費版每日自動更新一次、下拉更新需付費」，但推導後否決：

- 純粹記錄資產的人，更新頻率本來就低——一週甚至一個月一次。**每日一次對他們是隱形的上限。**
- 隱形的上限轉換不到任何人。真正會一天更新多次的是盯盤者，而盯盤者不是本 App 的目標用戶（他們用券商 App）。
- 結果兩頭落空：免費用戶覺得被刁難，該付費的人又沒被打中。

**即時報價是交易 App 的付費軸線，不是資產記錄 App 的。** 且「數字永遠是準的」應是資產記錄 App 的基本體驗，在此設卡傷害大於收益。

附帶效益：省下 `lastQuoteSyncAt` 狀態欄位、首次／更新抓價的路徑區分、批次更新 UI——第一階段工程量顯著下降。

**保留風險**：Yahoo API 為非官方、無 SLA，若未來遭 rate limit 或封鎖，需重新評估（屆時應考慮加長快取或改付費資料源，而非改為向用戶收費）。

### 決策五：保險為「純呈現、不計入淨值」的分類，欄位設計另開獨立 spec

`Insurance` model 已存在於 schema，但**全 codebase 無任何 UI**（僅 `landing-content.tsx` 行銷文案提及）。且現有欄位（`declaredRate`、`cashValueData`、`surrenderValue`…）是針對**儲蓄／投資型保單**設計的，無法涵蓋醫療、癌症、意外等其他險種。

定案的產品定位：

- 保險**不計入淨資產**。它不是資產也不是負債，而是「統計你有哪些保單」的呈現，不做專業精算分析。
- 透過既有的 `+` → `entry/new` 流程新增（`entry/new` 由 `categoryConfig.ts` 的 `CATEGORIES` 驅動，加一個「保險」top category 即可），新增體驗與其他分類一致。
- 技術上以 `Entry.includeInChart = false` 讓保險 entry 自動不進淨值圖表與總資產。
- 保險卡片顯示在 entry 列表（新主題色），點開卡片列出每筆保單，點進去是**獨立的保單詳情頁**。

**不新增 nav 分頁**——`BottomNav.tsx` 是四格浮動玻璃膠囊（`資產 / 收支 / 退休金 / ＋ / 設定`），加第五格會破壞版面；且走 `entry/new` 已能滿足新增需求。

**險種與欄位設計、詳情頁呈現，另開獨立 spec 處理**（涉及醫療／癌症／投資型／意外等多種險種，各自欄位不同，需專門 brainstorm）。本文件僅定義它的分層歸屬（Premium）與「不計入淨值」的定位。

### 決策六：退休試算維持免費

兩個理由：

1. **它已是 nav 主分頁**。在四個主 tab 之一設付費牆，用戶每次點都撞牆，體感是「這 App 到處要錢」。
2. **標準一致性**。一次性計算、不耗 API 成本、是良好的 SEO 獲客內容——與貸款試算同類，不應一個免費一個付費。

**保留的付費變體**：免費版是「手動輸入數字→算結果」的計算機；付費版**帶入真實資產、負債、貸款攤還**做推演。付費牆因此是加值而非擋路。此變體依賴決策七的推算引擎，屬後續階段。

### 決策七：淨值預測為第二階段主打

結合 `Loan`（攤還表）、`Insurance.declaredRate` 與 `cashValueData`（保價金表）、`Recurrence`（固定收支），推算未來 6/12/24 個月淨值曲線。

**這是最強的差異化**：市面記帳 App 算不出來，因為沒有貸款攤還與保單現金價值的資料。但工程量足以撐起獨立 spec，不納入第一階段。

## 第一階段實作範圍

只做兩項：

### 1. 資產筆數上限（20 筆）

`entitlements.service.ts` 是唯一的權限檢查點，目前寫死 `return true`。改為查詢 `Subscription`：

- 依 `deriveAppleAccountToken(clerkUserId)` 查 `Subscription`
- `status` 為 `active` 或 `grace_period` 且 `expiresAt` 未過期 → premium
- `expired` / `revoked` / 查無資料 → free

新增資產的 API（`entries.service.ts` 建立路徑）在寫入前檢查：非 premium 且該 `userId` 的 `Entry` 數已達 20 → 回傳明確的錯誤碼，前端據此導向 paywall。

**檢查必須在 server 端**，client 端的筆數判斷僅用於提前顯示提示，不可作為唯一防線。

### 2. 資產配置分析

純計算，不改 schema。以現有 `Entry.topCategory` / `subCategory` 聚合：

- 各類資產佔淨資產比重
- 集中度風險提示（例：單一持股佔淨資產 40%）
- 資產 / 負債結構比

選它作為第一階段的第二項，是因為它**不需要新 schema、不需要新資料來源**，卻立刻讓 Premium 有看得見的價值——單靠「20 筆上限」說服力偏薄。

### 明確排除

保單管理、多幣別損益、淨值預測、AI 資產問答、家庭共享資產表，均不在第一階段。

## 後續階段（不在本文件範圍）

依實作難度排序，各自需要獨立 spec：

1. **多幣別資產換算**——外幣資產換台幣、匯率損益（`exchange-rate` API 已存在）
2. **淨值月報**——以 `EntryHistory` 算「本月淨值變動 +12 萬，主要來自 X」，提供每月回訪的理由
3. **保單管理**——schema 已備，UI／CRUD 從零做
4. **淨值預測**——推算引擎（第二階段主打）
5. **退休試算帶入真實資產**——建於 4 的引擎之上
6. **AI 資產問答**——差異化強，但 API 成本需控管
7. **家庭共享資產表**——付費意願最高，但需將全部 query 從 `userId` scope 改為 household scope，屬全面性改動；**應等訂閱模式驗證成功後再評估**

## 付費牆文案（定案）

用於新增第 21 筆資產時的升級提示。選用原則：肯定使用者是重度用戶，避免「已達上限」的懲罰語氣（尤其那位 22 筆的既有用戶會第一個看到）。

- **標題**：你的資產版圖越來越豐富了
- **內文**：身為重度用戶，你值得更大的空間。免費版可記 20 筆，升級 Premium 解鎖無上限，輕鬆管理。
- **主按鈕**：解鎖無上限
- **次按鈕**：稍後再決定

## 訂閱設定需求（實作前置，僅 iOS）

**平台範圍**：僅 iOS IAP（透過 RevenueCat）。**不做 web 端訂閱**——web 目前確定不使用。

### App Store Connect

- **訂閱群組（Subscription Group）**：月繳／年繳放同一群組，Apple 自動處理升降級與退款差額，同群組內單一用戶僅一個生效訂閱。
- **產品**：設 Reference Name 與全域唯一 Product ID（如 `com.app.pro.monthly`），完成在地化名稱與描述。
- **首購／試用（Introductory Offer）**：Free Trial／Pay As You Go／Pay Up Front，同群組單一 Apple ID 預設限用一次。
- **定價**：只能選 Apple 固定價格點（Price Tiers），不可任意填數字；定價須與價值相稱，否則以欺詐性定價退件。台灣售價為含稅價。
- **抽成**：標準 30%；連續訂閱滿一年或加入小型企業方案降為 15%。

### RevenueCat 對應架構

程式端只驗證 Entitlement：

```
App Store Product ID → RevenueCat Package（週期）
  → RevenueCat Offering（Paywall 方案組合）
    → RevenueCat Entitlement（功能權限識別碼，如 pro_access）
```

### 送審必備

- **綁版本送審**：首個訂閱產品必須與一次 App 版本更新合併提交（Version 頁勾選加入），不可單獨送審。
- **審核附件**：上傳 Paywall 完整截圖；審核備註提供沙盒測試帳號與購買路徑。
- **Paywall 規範**：須明確標示訂閱名稱、價格、計費週期，並附「隱私權政策」與「使用條款（EULA）」可點連結。
